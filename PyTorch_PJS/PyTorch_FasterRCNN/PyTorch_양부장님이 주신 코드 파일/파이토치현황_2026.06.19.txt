제 생각에 가능성이 높은 순위는
① 현재 조합 자체의 버그 (가장 유력)
torch 2.7.1
torchvision 0.22.1
FasterRCNN
wrapper
trace
Global alloc not supported yet
② trace 방식이 맞지 않음

FasterRCNN은 trace보다 script가 더 적합한 부분이 있음.

③ torchvision 0.22.1의 문제

실제로 Detection 모델의 TorchScript 관련 버그가 여러 번 있었습니다.

제가 권하고 싶은 것은

현재 C++ 코드는 더 이상 건드리지 말고,

Python 환경을
torch 2.5.1 + cu118
torchvision 0.20.1 + cu118

정도로 낮춘 후,

wrapper + trace()

또는

wrapper + script()

를 다시 시도해 보는 것입니다.

그리고 개인적으로는,

지금까지 나온 증상으로 보아 현재의 .pt 생성부 자체에 문제가 있는지, 아니면 2.7.1/0.22.1 조합의 버그인지를 확인하기 위해

가장 단순한 FasterRCNNWrapper + trace() 전체 생성 코드를 처음부터 다시 구성해 보는 것이 필요해 보입니다.

제 의견

현재 사용 중인

PyTorch 2.7.1
TorchVision 0.22.1
LibTorch 2.7.1
FasterRCNN
RTX4070Ti

조합에서는,

순수 PyTorch만으로 할 수 있는 최적화는 이미 80~90% 정도 수행하셨습니다.

남은 문제인

Global alloc not supported yet
53ms ↔ 수 초의 변동
Warmup 후에도 불안정

은 코드의 문제가 아니라 FasterRCNN + TorchVision + TorchScript 구조적 특성에 가까운 문제입니다.

따라서 "순수 Torch 환경에서 절대 불가능"은 아니지만,

산업용 수준(±5% 이내)의 추론시간 안정성을 얻으려면 TensorRT 계열로 넘어가는 것이 가장 현실적인 방법이라고 볼 수 있습니다.
